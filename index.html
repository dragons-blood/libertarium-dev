🔧 Watchdog / Fixer
A background thread that sweeps the system every ~100s looking for trouble: stuck sessions, wedged gauntlet targets, zombie processes, stale lockfiles, Python tracebacks in session logs, and missing API keys. What it does about what it finds depends entirely on the mode you pick below.
loading…
ℹ️ What do these modes actually do?
⚫ OFF
Sweep thread is stopped. Nothing is detected, logged, or fixed. The watchdog is effectively not running. Use this if you want full silence.
🔎 COLD SWEEP  (default on startup)
Detects & logs only. Every sweep writes incidents to the audit ledger and fires toast notifications, but never takes any action — not even GREEN ops fixes. This is the safe observation mode: watch what it wants to do before you trust it to do it. Zero blast radius.
✅ SAFE AUTO  (recommended when you trust it)
GREEN lane auto-applies, YELLOW lane stages for your approval.
GREEN (auto): reversible operational fixes only —
  • Stop sessions idle for >180s (calls session.stop())
  • Mark zombie sessions (status=running but process dead) as ended
  • Abort gauntlet targets running >300s with 0 attempts
  • Delete stale lockfiles (>600s old, pid dead)
YELLOW (staged): code-level fixes —
  • Detects Python tracebacks in session logs
  • Spawns a locked-down fix-agent that reads only the evidence file
  • Fix-agent writes a single unified diff + explanation to state/watchdog_staging/
  • You review it in the Staged Fixes panel below and hit Apply or Reject
  • Nothing touches disk without your click. git apply runs only after Apply
Rate-limited: max 10 GREEN/hr, 5 YELLOW/hr, 1 fix per incident-kind per session. RED-zone files (server.py, fixer.py, secrets, .git/, state/) are categorically blocked from any fix, even if you hit Apply.
⚠️ AGGRESSIVE  (use with caution)
GREEN + YELLOW both auto-apply. Same rate limits, same RED-zone enforcement, same git apply --check validation — but you lose the human-in-the-loop approval step. The fix-agent writes a diff, the server validates it, and git apply runs immediately. You'll see it in the toast and the ledger, not in the Staged panel.
Use this when you've watched Safe Auto long enough to trust the fix-agent's judgment on a particular class of incident, or when you're afk and want self-healing without the approval bottleneck. Rollback watch (60s after a fix, auto-revert if a fresh crash appears) applies in both modes.
🐺 WOLVERINE  (agent doctor / self-healing)
Everything AGGRESSIVE does, PLUS autonomous agent diagnosis. Wolverine scans each running session's logs every ~15 turns and looks for pathological patterns:
  • Error loops — same error repeated 3+ times (hammering a broken approach)
  • Tool loops — same tool called 5+ times in a row (going in circles)
  • Idle drift — session stalled with no output for 2+ minutes
  • Refusal spiral — hitting content policy walls repeatedly
  • Analysis paralysis — reading endlessly but never writing/editing
  • API retry storm — rate limits or upstream errors piling up
When a pattern is detected, Wolverine sends the agent a targeted diagnostic message with specific guidance to break through the barrier — not killing the session, but helping it think differently. Like an agent doctor on call.
Rate limited: max 6 interventions/hr, 5-min cooldown per session, scan every 15 turns.
🚨 PANIC
Emergency stop. Kills the sweep thread immediately, writes a red-flagged ledger entry, flips the mode to OFF. Pending staged fixes stay on disk but nothing new will ever be detected or applied until you turn the watchdog back on. Use this if the watchdog itself starts misbehaving.
Always true, regardless of mode: RED-lane incidents (missing API keys, anything that would touch secrets or .git) are never auto-fixed — they're logged and you're notified, that's it. The RED zone enforcement is a second safety net inside apply_staged_fix(): even if a malicious diff ended up staged, the server will refuse to apply it.
